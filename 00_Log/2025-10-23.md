# Donde ingresar cambios en fpnew_pkg

Para permitir que se generen mas slices por formato deberiamos introducir un nuevo campo en la estructura fpu_implementation_t. Este tipo nos da control sobre las slices que se generaran dentro de cada opblock.

Este es el nivel mas bajo que podemos configurar, la cantidad de lanes (si es que tenemos los vectores activados) se determinan en tiempo de elaboracion usando funciones que basicamente dividien el ancho total de la FPU por el ancho de el fomato. **fpnew_pkg::num_lanes()**

Introducir un cambio en fpu_implementation_t va a implicar acomodar el cambio en todos los tipos de procesadores predefinidos, **pero deberiamos buscar manera de definir un valor por default dentro de la estructura**. Para el momento de implementacion podemos modificar uno de los predefinidos (menos trabajo pues no tendria que modificar las distintas configuraciones a nivel cva6) o definir un nuevo tipo a nivel cva6.

Como diferencio de la cantidad de slices que quiero generar en cada opblock? Puedo hardcodear en el generate block de fpnew_opgroup_block.sv tal que

``` verilog
if (OpGroup == fpnew_pkg::ADDMUL) begin : check_for_multiple_slices
    for(genvar j = 0; j < fpnew_pkg::num_slices; j++) begin : gen_multiple_slices
        fpnew_opgroup_fmt_slice #(...) copied_slice (...)
```

## Cosas a modificar en fpnew_opgroup_block.sv

Las cosas a modificar a grandes rasgos seran:

+ Redirrecion de input en caso de operacion en ADDMUL
+ Arbitrer de input entre multiples unidades de computo (prioridad fija/round-robin)
+ Arbitrer de output entre multiples unidades de computo (round-robin)
+ Modificacion de seniales de handshake (ready/valid) para multiples unidades
+ Modificar parametros de `fpnnew_top.sv` y `fpnew_opgroup_block.sv` para mandar nuevo parametro respetando previo disenio
+ Modificar `fpnew_pkg.sv` para incluir nuevo parametro
+ Modificar `fpu_wrap.sv` para incluir nuevo parametro
