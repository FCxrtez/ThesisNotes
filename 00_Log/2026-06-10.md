# reduction Testbench

Se implemento el testbench. Este imprime a consola de vivado, y al mismo tiempo hace un dump de las señales que miramos en un archivo generado como parte del tb, para esto se utiliza una task. Encontre errores en la clasificacion de los numeros ya que faltaban algunas señales de is_boxed, entonces quedaba como `x` y esto se esparcia mas adelante a casi todos los datos importantes. Todavia falta comprobar que la logica del modulo este haciendo lo que esperamos que haga.

Empece a cambiar el nombre de parametros -> mala idea
El parametro `NUM_LANES` -> ahora es `NUM_OPERANDS`. No sirve calcularlo a partir de `fpnew_pkg::num_operands`, esta funcion toma como parametro el opgroup, no representa la cantidad de datos que entran dentro del bus. Ademas creo que esta funcion se usa para instanciar cuantos buses se generan que van hasta la unidad de reduccion en modulos de mas alta jerarquia, tocar esto genera un cambio mas profundo de lo que creo.

Cambie el is_boxed_i para que tome 4 bits, ya que nosotros tenemos 4 operandos. Ahora un detalle importante, es que no creo que sea de valor para nostros el is_boxed ya qu se utiliza cuando viene un dato en un bus mucho mas ancho de lo que ocupa el formato del dato, nosotros siempre vamos a estar ocupando todo el bus de datos ya que los datos vienen packed. Pero este dato de is_boxed se utiliza en la clasificacion de los datos. Creo que los mas simple seria escribir nuestro propio modulo de clasificacion para que no se fije en ese parametro y no andar tocando el modulo que es usado en el resto de la FPU.

Por otro lado me di centa que tengo muchisimos puertos sin utilizar, deberia terminar con la implementacion y redefinir los puertos del modulo, muchos quedaran inutilizados, esto tira un warning de assignment cuando corro el linter.

El tb anda, falta realmente comprobar que la logica este haciendo lo mismo que nostros queremos. Veo dos caminos para tomar a partir de ahora:

- **Terminar de implementar el modelo de python primero**: Esto nos permitiria en un futuro concentrarnos completamente en el modulo de sv, y seria un proceso mucho mas rapido de hacer ya que debuguear sv lleva mas trabajo a mi parecer
- **Seguir con implementacion de system verilog**: Todavia queda bastante logica en el modelo de python para pasar al system verilog, y como estuve viendo ahora que desarrolle el modulo, quedan varios detalles todavia en la interfaz del modulo de reduccion y en como se instancia en la FPU a pulir. Lo malo seria que dejariamos el modelo de python incompleto, lo que nos perjudicaria mas adelante con los tb automatizados que tengo en mente, y ademas es un medio menos transparente que python, mas dificil jugar con la logica hasta obtener lo que quiero.

Ademas la implementacion esta bastante desprolija, **mezclamos constantemente la intencion de implementar un modulo parametrizable con algunas variables hardcodeadas.** Dudo tomarme el trabajo de implementar y verificar el modulo para otros formatos de FP, deberia encontrar un balance para que deje de ser tan heterogeneo el codigo.
